# AutoDoorBLE 长期开发纪律

本文件只记录长期有效的开发规则。具体功能、接口、参数和实现细节以 `README.md` 导航的中文模块文档为准。项目方向或工程约束发生长期变化时，应同步调整本文件。

## 开始开发前

1. 先阅读根目录 `README.md`，确认项目整体目标和模块导航。
2. 修改代码前，完整阅读所有对应模块文档的“总”和“分”两部分。
3. 涉及多个模块时，先确认模块间接口和调用顺序。
4. 检查当前 Git 改动，不覆盖或撤销与本次任务无关的用户修改。

## 代码与文档同步

1. 每个功能模块必须有对应的中文模块文档。
2. 代码的接口、状态、流程、参数、依赖或可观察行为改变时，必须同步文档的代码重建规格。
3. 根据文档修改或重建代码后，必须反向检查文档与实际代码是否一致。
4. 新增、合并、拆分或删除模块时，同步更新 README 的模块导航。
5. 文档概要可以简洁，但重建规格必须足以恢复结构和行为一致的代码。
6. `docs` 文件名使用两位数字表示推荐阅读层级；同级模块可使用相同数字，实际代码生成顺序以模块依赖为准。

## 实现纪律

1. 保持 `AutoDoorBLE.ino` 为薄入口，应用装配和生命周期放在 `AutoDoorApp`。
2. 主循环优先采用非阻塞 `update()` 调度；新增阻塞行为必须说明原因和影响。
3. 保持模块职责边界，Web、BLE 和 Wi-Fi 层不得私自实现门控决策。
4. 不在日志、API 或 BLE 响应中新增密码、令牌等敏感信息输出；发现已有泄漏应明确报告。
5. 不擅自改变 BLE UUID、REST 字段、状态字符串等外部契约。
6. 涉及舵机、传感器或关门逻辑时，硬件安全优先于功能便利。

## 验证与交付

1. 默认只做代码与文档的静态审查，不执行本地脚本、编译、烧录或测试命令；只有用户明确要求时才执行，并记录实际命令和结果。
2. 软件检查、固件编译、台架验证和整机验证必须分别说明。
3. 没有真实设备证据时，不得宣称硬件行为已经验证。
4. 交付时列出代码改动、文档改动、兼容性影响、已完成验证和仍需人工验证的项目。
