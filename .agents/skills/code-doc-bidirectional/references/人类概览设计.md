# 人类概览设计

## 目标

让第一次接触模块的人在不阅读代码重建规格的情况下，理解模块为什么存在、怎样参与系统运行、与谁协作以及使用时要注意什么。

概览不是缩短版 API 文档。不要简单罗列所有函数、成员和字段，也不要让每篇文档看起来像同一张表格替换了名词。

## 固定原则

1. 保留 `总：模块概要（给人阅读）` 标题和总/分分界。
2. 开头使用一到两段自然语言建立模块定位。
3. 说明模块在系统中的上游、下游或协作对象。
4. 只保留人需要提前知道的关键限制、安全条件和失败影响。
5. 函数签名、成员、精确算法和协议字段放到“分”部分。
6. 不强制所有模块使用相同小标题或相同图形。

## 按模块类型选择结构

| 模块类型 | 推荐表达 | 重点回答 |
|---|---|---|
| 入口与调度 | 系统全景图、启动时间线、主循环说明 | 系统怎样组合并开始运行 |
| 配置 | 分类树、分组表、影响等级 | 改什么会影响哪里，风险多大 |
| 硬件感知 | 数据流、标定场景、接线表 | 原始数据怎样变成上层结果 |
| 硬件执行 | 控制流、运动循环、机械限制 | 指令怎样变成实际动作 |
| 业务决策 | 现实场景、状态图、模式对比 | 系统为什么做出某个决定 |
| 网络生命周期 | 状态图、恢复流程 | 怎样扫描、连接、断线和恢复 |
| 协议交互 | 用户场景、时序图 | 数据在参与者之间怎样传递 |
| 用户界面 | 页面线框、操作流程 | 用户看到什么、能做什么 |
| API/适配层 | 请求流、能力表 | 外部请求怎样进入内部模块 |

一个模块可能同时属于两类。选择最能建立理解的主结构，不要把所有推荐表达全部使用。

## 图形选择

- 三个以上模块或方向关系复杂：使用 Mermaid flowchart。
- 状态随事件转换：使用 stateDiagram。
- 多参与者按时间交互：使用 sequenceDiagram。
- 页面区域和布局：优先使用简洁文本线框。
- 简单一进一出关系：使用单行文本箭头，不必 Mermaid。
- 参数对照与风险分类：使用表格或文本树。

只在图形比两三句话更容易理解时使用。图中节点使用读者熟悉的业务名称，避免私有成员、局部变量和内部函数名。

## 推荐组合

通常选择以下组合之一：

1. 自然语言定位 + 系统关系图 + 运行说明。
2. 用户场景 + 时序图 + 职责分工。
3. 现实场景 + 状态图 + 模式说明。
4. 数据流图 + 标定/过滤说明 + 使用限制。
5. 页面线框 + 用户操作流 + 安全提示。
6. 分类树/表格 + 修改影响 + 生效方式。

## 避免的写法

- 每篇固定使用“一句话定位、负责什么、输入输出、工作流程、职责边界”。
- 同一事实同时在段落、列表和表格中重复。
- 把完整 API 路径、私有变量和逐行算法放进概览。
- 为只有两个节点的简单关系绘制大型图。
- 图形很好看但没有帮助读者回答实际问题。

## 完成检查

- [ ] 不读“分”部分，也能说出模块的用途和系统位置。
- [ ] 读者能理解一个典型运行场景或主要数据流。
- [ ] 图形与文字互补，没有重复表达全部内容。
- [ ] 概览没有泄漏过多代码级细节。
- [ ] 结构符合模块特点，而不是机械套模板。
