# 模块划分规则

## 优先依据

1. 共同实现一个明确业务或技术职责。
2. 具有稳定的公开边界和较高内部内聚性。
3. 生命周期、状态和配置由同一组件管理。
4. 文件通常一起修改、测试或替换。

## 常见合并

- 同一个类或组件的声明与实现。
- 薄入口与其应用装配类。
- 单页前端的结构、样式和脚本。
- 同一功能目录下不可独立理解的辅助文件。

## 常见拆分

- 独立协议适配器与核心业务。
- 持久化层与领域逻辑。
- 可独立替换、测试或部署的组件。
- 拥有不同外部接口或安全边界的组件。

## 控制复杂度

小项目默认使用单层 `docs/`。只有模块数量明显增加或存在清晰领域边界时才建立子目录。不要为每个源文件机械创建文档，也不要把整个项目塞进一篇无法维护的文档。
