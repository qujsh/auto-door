# 代码文档双向工程

## 选择工作模式

根据用户意图选择一种模式：

1. 代码生成文档：扫描代码并生成或更新 README、AGENTS.md 和模块文档。
2. 文档生成代码：根据 README、AGENTS.md 和模块重建规格生成项目。
3. 双向同步：根据用户指定的权威侧更新另一侧。
4. 一致性审计：只报告差异，不修改文件。

如果代码与文档冲突且用户没有指定权威侧，先列出影响行为的差异并请求确认。格式、注释或行为等价的内部重构不必制造虚假冲突。

## 代码生成文档

1. 检查仓库规则和未提交改动。
2. 识别技术栈、入口、构建方式、外部接口和运行边界。
3. 按职责和内聚性划分模块，不机械地每文件生成一篇文档。头文件与实现、入口与装配、强耦合页面资源可以归为一个模块。
4. 读取 [模块划分规则](references/模块划分规则.md)。
5. 按 [README 模板](references/README模板.md) 生成项目概览、模块导航和后续计划。
6. 按 [AGENTS 模板](references/AGENTS模板.md) 提取长期纪律，不写入易变的功能细节。
7. 读取 [人类概览设计](references/人类概览设计.md)，根据模块类型选择叙述、场景、流程或图形表达；再按 [模块文档模板](references/模块文档模板.md) 生成中文文档。
8. 使用 [完整性检查表](references/完整性检查表.md) 检查能否从文档重建代码。

### 文档文件优先级

- 使用两位数字前缀：`00-模块名称.md`、`10-模块名称.md`。
- 数字表示推荐阅读层级，不代替代码依赖图。
- 同级且互不依赖的模块使用相同前缀，例如两个底层驱动均为 `20-`。
- 同层存在明确依赖时使用相邻编号，例如基础服务 `30-`、依赖它的适配服务 `31-`。
- 默认按 10 预留编号，便于后续插入模块。
- 代码生成顺序必须根据文档依赖关系计算，不能只按文件名排序。

## 文档生成代码

1. 完整阅读 README、AGENTS.md 和全部模块文档。
2. 从每份文档的“文件映射”建立依赖图和生成顺序。
3. 检查技术栈、路径、类型、公开接口、内部状态、算法、错误处理和验收条件是否足够。
4. 对关键缺失信息停止猜测并提出问题；可安全推导的实现细节要明确标为推导。
5. 先生成公共类型和底层模块，再生成业务模块、适配层、入口和装配。
6. 编译和测试；把发现的规格缺口回写文档。
7. 目标是结构与行为一致，不要求逐字符、空格、注释或局部变量完全一致。

## 双向同步与审计

- 从改动文件定位拥有该文件的模块文档。
- 比较文件路径、依赖、类型、签名、默认值、状态机、算法、错误处理、日志、协议和不变量。
- 代码为权威时更新文档；文档为权威时更新代码。
- 新模块同时新增中文文档和 README 导航。
- 删除模块同时清理映射和导航，但保护无关用户改动。

## 输出原则

- 默认使用中文文档和中文沟通；代码标识符保持项目语言习惯。
- 模块文档必须有醒目分界：“总：模块概要（给人阅读）”与“分：代码重建规格（给 AI 或修改代码的开发者阅读）”。
- 概要追求易懂，但不强制所有模块使用相同小标题。根据模块特点选择全景图、数据流、状态图、时序图、生命周期、页面线框、场景叙述或影响表。
- 只在图形能明显降低理解成本时使用 Mermaid；简单关系使用短段落、表格或文本图即可。
- 避免每篇机械重复“一句话定位、负责什么、输入输出、职责边界”。这些信息可以自然融入更符合模块特点的结构。
- 重建规格保持统一，追求完整、确定、可验证，不因人类概览的自由结构而省略技术细节。
- 交付时报告工作模式、映射变化、重建等级、验证证据和信息缺口。
